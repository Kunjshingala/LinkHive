# Clean Architecture Refactor Plan for LinkHive

## Summary

Refactor LinkHive from a feature-first layered Flutter app into a stricter clean-architecture shape with clear `presentation -> domain -> data` boundaries, canonical domain models, repository interfaces, use cases, and infrastructure-only framework access.

The refactor should preserve current behavior first, then fix the architectural gaps found in review:
- remove framework/global singleton access from repositories and blocs
- stop widgets from performing business/data work directly
- replace localized persistence values with canonical domain values
- formalize offline sync and delete flows
- make `.ai` guidance match the real codebase and target architecture

## Target Architecture and Runtime Flows

### Target layers

- `presentation/`
  - screens, widgets, blocs/cubits, view models, route wiring
  - depends only on `domain`
- `domain/`
  - entities, value objects, repository contracts, use cases
  - pure Dart, no Flutter/Firebase/Hive imports
- `data/`
  - repository implementations, DTOs/models, mappers, local/remote datasources
  - owns Hive/Firebase/HTTP/share-intent integrations
- `core/`
  - DI, theme, localization, routing shell, common result/error types, app coordinators

### Mandatory domain contracts to introduce

- `AuthRepository`
- `LinkRepository`
- `CategoryRepository`
- `SyncRepository` or fold sync into `LinkRepository` with explicit sync methods
- `SettingsRepository`
- `SessionRepository` or `SessionProvider`
- `MetadataRepository`
- `ShareIntentRepository`

### Mandatory domain types to introduce

- `Link`
- `Category`
- `LinkPriority` enum with canonical values `high`, `normal`, `low`
- `SyncState` or `PendingOperationType`
- `Failure` hierarchy
- query/filter object for home list, e.g. `LinkQuery`

### App flows after refactor

- App bootstrap:
  - `main` initializes Firebase/Hive, registers datasources/repositories/use cases, starts app coordinator
  - app coordinator starts share-intent listener and connectivity listener
  - presentation reads state through use cases only
- Auth flow:
  - UI -> `AuthBloc` -> auth use case -> `AuthRepository`
  - session stream exposed through repository/provider, not `FirebaseAuth.instance` calls in bloc/repository
- Add/edit link flow:
  - UI -> `AddLinkBloc`
  - bloc holds all form state including priority/category selection
  - save -> use case -> repository contract -> local datasource first -> pending sync queue -> remote sync when eligible
- Home/list flow:
  - UI -> `LinkBloc`
  - bloc stores current query/filter/pagination state
  - repository emits watch stream of canonical entities
  - external changes re-query with existing filters preserved
- Sync flow:
  - connectivity/session event -> sync coordinator/use case
  - process pending upserts and pending deletes
  - pull remote state if authenticated
  - merge through repository rules, not directly from widgets/blocs
- Category flow:
  - category add/delete goes through category use cases
  - deleting category also updates affected links or applies explicit orphan-handling policy

## Implementation Plan

### Phase 1: Freeze architecture decisions and align docs

- Update the architecture target in `.ai/context/architecture.md` to the new clean-architecture structure.
- Update `.ai/conventions/code-style.md` so it allows `StatefulWidget` only for ephemeral UI state and forbids widget-to-repository calls.
- Keep `flutter_bloc`, `go_router`, `get_it`, Firebase, Hive, and current UI design system.
- Treat generated files as out of scope for manual edits except when regenerated by tooling later.

### Phase 2: Extract domain layer

- Create a pure Dart `domain` package/folder under each major bounded context or under `lib/domain/`.
- Move business rules out of current Hive/Firestore models into domain entities.
- Replace string priority values with `LinkPriority`.
- Introduce use cases:
  - `SignInWithEmail`
  - `SignUpWithEmail`
  - `SignInWithGoogle`
  - `SignOut`
  - `GetCurrentSession`
  - `ObserveSession`
  - `LoadLinks`
  - `LoadNextLinksPage`
  - `SearchLinks`
  - `SaveLink`
  - `UpdateLink`
  - `DeleteLink`
  - `SyncLinks`
  - `ObserveLinks`
  - `LoadCategories`
  - `AddCategory`
  - `DeleteCategory`
  - `LoadSettings`
  - `ChangeTheme`
  - `ChangeLocale`
  - `FetchLinkMetadata`
  - `ConsumeSharedLink`
- Add `Failure` mapping so blocs receive typed failures instead of raw SDK exceptions.

### Phase 3: Rebuild data layer around contracts

- Split current `LinkModel` and `CategoryModel` into:
  - domain entities
  - local DTOs for Hive
  - remote DTOs/maps for Firestore
- Replace `lib/features/links/repository/link_repository.dart` with repository implementations that depend on datasources and a session provider, never `FirebaseAuth.instance`.
- Introduce datasources:
  - `HiveLinkLocalDataSource`
  - `HiveCategoryLocalDataSource`
  - `FirestoreLinkRemoteDataSource`
  - `FirestoreCategoryRemoteDataSource`
  - `FirebaseAuthDataSource`
  - `LinkMetadataRemoteDataSource`
  - `ShareIntentDataSource`
  - `SettingsLocalDataSource`
- Add mappers between DTOs and domain entities.
- Add pending operation storage for delete and update reconciliation.
- Define merge policy explicitly:
  - local writes always persist first
  - pending operations are queued
  - remote pull does not silently resurrect locally deleted records
  - category delete updates affected links using a single repository policy

### Phase 4: Refactor presentation layer

- Keep one bloc per screen where useful, but move all business decisions into use cases.
- Refactor `AddLinkScreen` so category list, selected categories, priority, metadata state, and validation all live in bloc state.
- Refactor `HomeScreen` so it does not directly know repository details; it only dispatches filter/query/pagination events.
- Preserve active filters/search/pagination on external updates.
- Refactor auth/account screens to stop calling repositories from UI code.
- Introduce a route-level auth/session guard instead of comments or planned `AuthGate`.
- Standardize bloc states to expose:
  - view data
  - loading flags
  - error/message effects
  - one-shot navigation actions if needed

### Phase 5: Refactor app infrastructure

- Rebuild `locator` registration by layer:
  - datasources
  - repository implementations bound to domain contracts
  - use cases
  - blocs/cubits
  - app coordinators
- Move `ReceiveSharedIntent` and `SyncService` out of ad hoc app bootstrap usage into coordinators that call use cases.
- Update routing so route names are semantic and path constants are separate; support typed extras for add/edit link.
- Keep theme/localization cubits for now, but make them use `SettingsRepository`.

### Phase 6: Fix edge cases during the architecture move

- Canonicalize priority values so localization only affects labels, never persisted values.
- Add category normalization rules:
  - trim
  - duplicate-name handling policy
  - case policy
- Add pending delete queue for offline delete.
- Add safe metadata fetch behavior for invalid URLs, unsupported schemes, and timeouts.
- Add share-intent handling for duplicate open events and malformed payloads.
- Add app-resume/session-change sync trigger policy.
- Add explicit guest-to-authenticated transition rules for locally created content.

### Phase 7: Testing and acceptance hardening

- Replace placeholder repository tests with real unit tests against contracts and fake datasources.
- Add mapper tests for domain <-> local/remote conversions.
- Add bloc tests for preserved filters, form state, one-shot errors, and sync-triggered reloads.
- Add widget tests for real screen behavior instead of scaffold placeholders.
- Add coordinator tests for share intent and connectivity/session-driven sync.
- Keep `fvm flutter analyze` and full `fvm flutter test` green after each phase.

## Public Interfaces / Types That Will Change

- `LinkModel.priority: String` -> domain `Link.priority: LinkPriority`
- repository access from concrete class -> abstract contracts
- direct `FirebaseAuth.instance.currentUser` reads -> `SessionRepository` or `SessionProvider`
- `AddLinkState` and `LinkState` should expose canonical domain values, not UI-localized strings
- route payload for add/edit link should become a typed argument object instead of raw `String? extra`
- sync API should expose explicit methods for:
  - queue operation
  - flush pending operations
  - pull remote updates
  - observe sync state

## Assumptions and Defaults

- Keep current libraries: `flutter_bloc`, `go_router`, `get_it`, Firebase Auth, Firestore, Hive.
- Do not change UI language/design system in this refactor except where needed to support architectural cleanup.
- Clean architecture means strict dependency direction, not a separate Dart package split.
- Generated files are not edited manually in the first refactor pass.
- Priority persistence will use canonical enum values; localization stays presentation-only.
- Offline delete support is required in the target architecture.
- Category delete will remove the category reference from affected links by default.
- This is a staged refactor; behavior parity comes before feature expansion.
